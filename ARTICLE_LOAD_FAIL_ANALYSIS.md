# 기사 목록→수정 화면 “다시 로그인” 오류 원인 및 조치

## 증상

- 대시보드 목록에는 기사가 보임.
- 제목 클릭 후 `article-write.html` 에서 상세 로드 실패 시, 과거에는 모든 실패를 **“저장된 기사 데이터를 불러오지 못했습니다. 다시 로그인…”** 한 가지로 처리.

## 왜 404 화면이 아니라 그 문구였는지

- 정적 파일 `article-write.html` 은 **200으로 열림** (이전 404는 `article-write` 확장자 누락 href — 별도 수정됨).
- 페이지 내부 `fetch` 가 실패하면 **브라우저 404 페이지가 아니라** 스크립트가 띄운 `alert` 만 보임.
- 상세 `GET` 이 실패한 뒤 **목록 폴백 `GET /articles` 도 배열로 파싱되지 않거나**, 목록에서 동일 id 를 찾지 못하면 위 통합 메시지 분기로 들어감.

## 핵심 원인 (운영)

1. **조회 경로 불일치**  
   - 대시보드: `GET /api/articles` (브라우저 → **Vercel 프록시** → Render).  
   - 과거 작성 페이지: `GET https://newswindow-backend.onrender.com/api/articles/:id` (**브라우저 → Render 직접**).  
   Render 쪽에서 **로컬 `articles.json` + 다중 인스턴스/비균등 디스크** 가 있을 때, 프록시 경로와 직접 경로가 **서로 다른 스냅샷**을 읽으면: 목록에는 있는데 상세만 없음(404) → 폴백 목록도 직접 호출이면 계속 불일치 → 최종 알림.

2. **에러 처리 뭉개짐**  
   - 401/403/404/파싱 실패/네트워크 오류를 구분하지 않고 동일한 “다시 로그인” 문구로 안내.

## 조치 요약

### `nw-office/article-write.html`

- **`API_READ`**: 로컬이 아니면 항상 **`/api`** (대시보드와 동일).
- **`API_WRITE`**: 운영 도메인만 Render 직접 유지 (대용량 POST/PATCH 4.5MB 제한 회피).
- `loadArticle`: 상세·목록 폴백 모두 **`API_READ`** 사용.
- 상태코드별 메시지: **401** 만 로그인 재안내, **403** 권한, **404** 없음, 그 외·파싱 실패는 서버/데이터 오류.
- 목록 폴백: `!listRes.ok` 또는 JSON 배열이 아니면 **“기사 데이터를 불러오는 중 오류”** 로 구분.
- 디버그: URL 에 `?nwdebug=1` 또는 `localStorage.nwOfficeDebug=1`.

### `nw-office/reporter.html`

- 동일 디버그 플래그로 제목 클릭 시 `data-id`·이동 URL 로그.

## 재현·확인

1. 기사 작성 → 작성완료/송고 → 대시보드에서 제목 클릭 → 수정 폼에 제목·본문 로드되는지.
2. 문제 시 `?nwdebug=1` 로 콘솔에서 `API_READ`·요청 URL·토큰 유무 확인.

## 회귀

- 대용량 **저장** 여전히 Render `API_WRITE` 로만 전송 (Vercel 413 방지).
