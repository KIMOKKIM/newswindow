# 기사 유실·덮어쓰기 감사

## 저장소

- 단일 JSON 배열(경로: `NW_ARTICLES_JSON_PATH` 또는 `backend/data/articles.json`).
- 메인·공개 API·대시보드 API 모두 **같은 in-memory `articles`** 을 읽음(코드 경로 상).

## 유실 정의

- **진짜 유실**: 배열에서 `id` 행이 삭제되거나, 배열 전체가 이전 백업으로 **통째 교체**됨.
- **가상 유실**: 행은 있는데 **기자 대시보드만** `author_id` 매칭 실패로 안 보임(메인에는 노출).

## 이번에 확인한 것

- 운영에서 **목록의 모든 id**가 상세에서 열림 → **“목록 id는 존재”**.
- 로컬/원격 샘플에서 **`author_id` 없는 published** 행 존재 → 기자 목록과의 **불일치 가능**.

## overwrite 가능 지점(코드)

- `articlesDb.save()` 가触는 모든 경로(작성·수정·승인·반려·조회수 증가 등).  
  잘못된 일괄 시드·수동 JSON 편집·디스크 복구 시 **배열 전체 교체** 시 유실로 보일 수 있음.

## 캐시

- `public/list` 는 Vercel 헤더로 `no-store` 설정됨.  
- `public/:id` 에 **no-store** 추가(배포본)로 중간 캐시로 인한 목록·상세 시점 차 완화.
