# 모달·대시보드 불일치 — 직접 원인 (확정 범위)

## 원인 후보 중 실제 코드로 확정된 것

1. **메인 리스트는 submitted 포함, 상세는 published 만**  
   → **현재 코드에서는 해당 없음.** 목록과 공개 상세 모두 **`isPublicFeedReadableStatus`** 를 쓴다.

2. **`author_id` 없는 공개 기사 + 기자 전용 API**  
   → **해당 있음(수정 전).**  
   - 목록·모달(공개 API)에는 나올 수 있음.  
   - 기자 **`GET/PATCH /api/articles/:id`** 는 `authorIdForArticle === null` 이면 **404** 로 끝나, 목록과 불일치할 수 있었음.  
   - **수정 후**: `reporterOwnsArticleRecord` 로 목록·`findById`·`update`·라우트 소유 검사를 **같은 규칙**으로 맞춤.

3. **모달 403 “공개된 기사만…”**  
   → 해당 id 의 **저장 상태가 공개 피드 밖**(draft/rejected 등)이거나, **클라이언트가 가진 목록 id 와 서버 상태의 시점 차**일 때 발생.  
   운영에서 list 의 **모든 id** 에 대해 detail 을 순회 검사하면 “목록 대비 상세 불일치”를 **즉시** 확인할 수 있다.

4. **편집장 목록에서 동시에 사라짐**  
   - `all()` 필터가 없으므로, **같은 저장소·같은 성공 응답**이면 코드만으로는 “전부 필터에 걸려 숨김”이 성립하지 않는다.  
   - **다른 API 베이스**, **인증 실패 후 빈 배열**, **파일 누락** 등을 병행 조사한다.

## 이번에 한 가지로 요약하는 “직접 원인”

**기자 대시보드·기사 상세(인증 API)가 `author_id` 누락 레코드를 `404`로 거부하던 것이, 메인·공개 API와의 체감 불일치를 만들었다.** (공개 모달 403 은 별도로 **상태 미충족** 또는 **캐시/시점**을 의심.)
